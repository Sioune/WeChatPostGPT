# 从一个想法到智能公众号排版体：WeChatPostGPT 的开发复盘

很多人写公众号时都会遇到同一个问题：内容已经写好了，但排版还要花很久。标题怎么放、段落怎么分、图片怎么处理、引用怎么突出、整体颜色是否舒服，这些事情看起来只是“美化”，其实会直接影响读者愿不愿意继续读下去。

WeChatPostGPT 就是围绕这个痛点做出来的。它的目标很简单：输入文章标题和正文，让 AI 自动完成一套美观、统一、能直接复制到微信公众号后台的排版。

这篇文章不是技术说明书，而是一篇开发复盘。它记录了这个工具是怎么从一个粗糙想法，一步步被 Codex 协助打磨成现在这个样子的。

## 一开始：不只是“分段”和“换颜色”

最初的需求并不是做一个普通排版器。普通排版器通常只是帮你分段、套标题、换几组配色。但公众号文章真正好看的版式，往往不是几个零散样式的组合，而是一整套视觉系统。

比如一篇活动文章，可能需要更醒目的海报式文头、清晰的时间线、强烈的行动引导；一篇深度分析文章，则需要克制的标题层级、舒适的行距、数据和引用的稳重呈现。

所以项目一开始就确定了方向：不是让 AI 随便修饰几段文字，而是让它理解文章气质，再选择或生成一整套模板。

## Codex 做的第一件事：先读懂，再动手

Codex 在这个项目里的角色，不是简单地“帮忙写代码”。它更像一个能进入项目现场的开发搭档。

它会先查看已有文件，理解页面结构、前端逻辑、服务端接口和样式布局，然后再决定改哪里。每次开发新功能前，它都会先搜索相关代码位置，比如输入框在哪里、模板库怎么保存、AI 接口怎么调用、预览结果怎么生成。

这让开发过程不像“凭空生成一段代码”，而更像真实工程里的迭代：先理解现状，再小步修改，再运行检查。

## 第一阶段：做出能用的智能排版

项目最早实现的是“AI 决策 + 本地渲染”。

这个思路可以理解为：AI 不直接写最终 HTML，而是先判断文章类型、阅读节奏、适合的模板和配色，再把这些决策交给本地渲染器。本地渲染器负责输出微信公众号更容易接受的内联 HTML。

这样做的好处是稳定。公众号后台对 HTML 支持有限，复杂脚本、外部样式、动画和很多网页能力都不适合直接复制进去。本地渲染器可以把输出控制在安全范围内。

但后来也发现了它的限制：稳定是稳定，创意却容易被框住。AI 只能在既定模板和字段里发挥，美感上限取决于本地渲染器有多强。

## 第二阶段：补齐真实产品会遇到的问题

真正使用时，问题很快出现了。

首先是 AI 配置。用户不应该改代码才能填 API Key，所以后来在页面右上角加入了 AI 配置菜单，可以填写 BaseUrl、API Key，测试连接，并自动拉取模型列表。

然后是图片。公众号排版里图片非常重要，但如果把上传图片直接变成 base64 塞进正文框，编辑体验会非常糟糕。于是项目改成把图片保存到本地素材库，正文里只保留短引用。这样图片可以参与排版，但不会把正文输入框撑成一团难读的代码。

再后来是模板库。用户会学习很多模板，所以必须能搜索、筛选、删除。模板不再是一次性结果，而变成一个可以积累的本地资产。

这些功能都不是“炫技”，而是产品真正可用所必须经历的打磨。

## 第三阶段：模板学习，从“像”到“真的学到”

模板学习是这个项目里最关键的一步。

用户希望输入一个参考文章 URL，或者粘贴一篇已经排版好的文章，让 AI 学习它的视觉风格，再生成可复用模板。

一开始，这个功能踩过坑。比如微信公众号 URL 抓取后，看起来好像成功了，但实际提取到的内容可能不是真正的正文区域，或者只是页面的一小段。这样学出来的模板就会和参考文章完全不一致。

这个问题被认真修正了：后端现在会优先提取微信公众号正文区域 `#js_content`，并返回提取模式、正文长度、样式数量等诊断信息。如果抓不到真实正文，就停止学习并提示用户，而不是假装成功。

这也是 Codex 在开发中很重要的一点：它不只是把报错“绕过去”，而是把问题拆开验证。抓到了什么？是不是目标正文？样式数量对不对？如果不可靠，就应该明确告诉用户。

## 第四阶段：升级模板 schema 和渲染器

为了让 AI 学到的不只是颜色和字号，项目把模板升级成 v2 结构。

新模板会描述六类核心版式：

- 文头怎么呈现
- 标题系统怎么设计
- 引用和金句怎么突出
- 列表是卡片、清单还是时间线
- 图片默认怎么摆放
- 分隔线和节奏如何收束

这样一来，AI 学到的不是“标题用蓝色”，而是“这篇文章适合一种什么样的阅读体验”。

本地渲染器也随之升级。它不再只按文章类型套固定模板，而是根据这些版式令牌生成整体效果。

## 第五阶段：加入 AI 原生创意排版

后来又发现一个关键差异：同样使用强模型，在 ChatGPT 里直接要求它“尽情发挥，生成可复制到公众号的富媒体成果”，效果往往更大胆、更漂亮。而项目里的 API 输出却显得拘谨。

原因很简单：项目原来的架构把 AI 管得太紧。它只能返回结构化模板参数，不能直接生成最终 HTML。

于是项目新增了一个模式：AI 原生创意排版。

这个模式下，AI 可以直接生成完整的公众号内联 HTML。它不再受本地模板枚举限制，可以自由设计文头、强调块、色彩节奏、分隔结构和整体视觉层次。

但自由不代表没有边界。服务端仍然会清洗危险内容，比如脚本、事件属性、外部样式、复杂定位等，确保结果更适合复制到公众号后台。

现在工具里就有两条路线：

- 稳定路线：AI 决策 + 本地渲染，适合模板复用和长期一致性。
- 创意路线：AI 原生创意排版，适合追求更强审美和内容贴合度。

## Codex 在这个过程中的价值

这次开发最直观的感受是：Codex 不只是一个回答问题的 AI，而是一个能参与完整开发闭环的工程助手。

它能做这些事：

- 阅读项目文件，理解已有结构。
- 根据需求修改前端、后端和样式。
- 调试 API 返回格式，定位是模型没返回，还是程序没解析。
- 设计更合理的错误提示，避免产品“假成功”。
- 运行本地检查，确认代码语法没坏。
- 启动本地服务，验证接口和页面行为。
- 初始化 Git 仓库，提交代码并推送到 GitHub。

更重要的是，它可以跟着需求变化持续迭代。这个项目不是一次写完的，而是在不断反馈中长出来的：图片太占正文，就改图片素材库；URL 学习不准，就做真实提取验证；模板太拘谨，就加入原生创意模式；页面被长文撑高，就改成固定框内滚动。

这种协作方式很接近真实产品开发。

## 最后的成品是什么

现在的 WeChatPostGPT 已经具备一套比较完整的工作流：

用户输入文章，选择排版引擎，AI 分析内容并生成排版；右侧展示微信兼容预览；满意后复制富文本，直接粘贴到公众号后台。

如果用户看到喜欢的文章模板，也可以粘贴富文本、输入 URL 或上传 HTML，让 AI 学习成自己的模板库。模板可以搜索、筛选、删除，也能被后续文章复用。

它仍然是一个本地原型，不会自动登录公众号，也不会替用户直接发布文章。但它已经把“写完文章后还要痛苦排版”的过程，压缩成了一个更轻松的智能工作台。

## 这次开发带来的启发

AI 工具真正有价值的地方，不只是“一次生成结果”，而是能参与一个产品从粗糙到可用的过程。

对普通用户来说，Codex 的意义也许可以这样理解：你不需要一开始就把需求说得完美，也不需要知道所有技术细节。你可以先说出一个目标，然后在使用中不断指出哪里不对、哪里不好用、哪里不像真实产品。

Codex 会把这些反馈变成具体的代码、界面、接口和流程。

这也是 WeChatPostGPT 的开发过程最有意思的地方：它不是单纯“写了一个排版工具”，而是展示了一种新的产品开发方式。

人提出审美和体验目标，AI 协助把它们一步步落成产品。
